# DNS-1 (slice 1/3) — `v2e-tf` DNS node + VyOS firewall/DHCP — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a dedicated `dns` node (`10.1.0.53`, mgmt VLAN, 1 vCPU / 512 MB / 8 GB, Debian template 9002) to the Terraform lab, plus the VyOS firewall rules that let the node VLANs reach it on `53/udp+tcp` and a VyOS DHCP server handing `option 6 = 10.1.0.53` to the node VLANs.

**Architecture:** The `dns` node joins the existing `local.nodes` map, so the current `for_each` machinery (cloud-init snippet, VM resource, SSH mesh, outputs) picks it up with no new resources. Two wrinkles are handled by extending the node schema with *optional* per-node overrides: `host_octet` (so `dns` lands on `.53` not `.10`) and `cores`/`memory`/`disk` (so `dns` gets 1/512/8 while the other three nodes keep their global defaults byte-for-byte). The VyOS changes are added to `cloud-init/vyos-router.yaml.tftpl`, fed by new locals in `router.tf`.

**Tech Stack:** OpenTofu 1.12.x, `bpg/proxmox` provider, VyOS 1.4/1.5 cloud-init (`vyos_config_commands`), templatefile.

**Source spec:** `v2e-docs/specs/2026-06-30-dns-appliance-design.md` (§2, §3, §7, §12).

**Environment note:** Proxmox is NOT reachable from the implementation environment, so `tofu plan`/`apply` cannot run here. In-env verification uses `tofu fmt -check` + `tofu validate` (offline: the provider is already in `.terraform.lock.hcl`) + `grep` on the rendered templates. `plan`/`apply` and the live `dig`/DHCP checks run on a Proxmox host / in CI.

## Global Constraints

- **Repo / branch:** `v2e-tf`, branch `feat/tf-dns-node`. Lands via **PR** (the `main` ruleset requires PRs) — do NOT merge to `main` locally. This is **PR #1** of three (tf → ansible → compose).
- **Node identity (exact):** name `dns`, `vm_id = var.node_vmid_base + 4` (= 314 with the default base 310), `template_id = var.debian_template_id` (9002), `octet = 0`, `vlan = 100`, `role = "member"`, `host_octet = 53`, `cores = 1`, `memory = 512`, `disk = 8`.
- **IP is derived, never hardcoded:** `10.1.0.53` = `${network_prefix}.${octet}.${host_octet}` = `10.1.0.53`. Do not write the literal IP into `network.tf`/`nodes.tf`; only `router.tf` locals reference `local.node_ip["dns"]`.
- **No behavior change for control/services/agent:** after this slice, a `tofu plan` against an existing deployment must show **only** additions for the `dns` node + the new VyOS config — the other three VMs must be untouched (same cores/memory/disk/IP). This is why the existing nodes get explicit `host_octet = var.node_host_octet`, `cores = var.node_cores`, etc. — the values are identical to today.
- **VyOS additions are guarded by `firewall_enabled` where they are firewall rules** (they live inside the existing `%{ if firewall_enabled ~}` block). The DHCP server is unconditional (new infra).
- **DNS node egress:** it must reach the internet for DoT (853) + the Technitium install. This is already covered by the existing forward rule 20 (`LAN -> internet`, source `lan_supernet = 10.1.0.0/16`), which contains `10.1.0.0/24`. Confirm, do not duplicate.

## Task 3: VyOS firewall rules for the internal resolver

**Files:**
- Modify: `router.tf:6-46` (add `dns_ip` to the templatefile call)
- Modify: `cloud-init/vyos-router.yaml.tftpl` (forward rules 32/40/41)

**Interfaces:**
- Consumes: `local.node_ip["dns"]`, `local.subnets`, existing `control_subnet/services_subnet/agent_subnet` template vars.
- Produces: template var `dns_ip`; VyOS forward rules allowing control→dns and the node VLANs→dns:53.

- [ ] **Step 1: Pass `dns_ip` into the vyos templatefile**

In `router.tf`, inside the `templatefile(...)` map (after the `agent_subnet` line, ~line 34), add:

```hcl
    dns_ip          = local.node_ip["dns"]
```

- [ ] **Step 2: Add the DNS forward rules to the VyOS template**

In `cloud-init/vyos-router.yaml.tftpl`, immediately **after** rule 31 (the `control -> agent` block ending at line 106) and **before** the closing `%{ endif ~}` (line 107), insert:

```yaml
  - set firewall ipv4 forward filter rule 32 description 'control -> dns node'
  - set firewall ipv4 forward filter rule 32 source address '${control_subnet}'
  - set firewall ipv4 forward filter rule 32 destination address '${dns_ip}'
  - set firewall ipv4 forward filter rule 32 action 'accept'
  # internal resolver — node VLANs may query the dns appliance on 53 (udp+tcp)
  - set firewall group network-group LAB-NODES network '${control_subnet}'
  - set firewall group network-group LAB-NODES network '${services_subnet}'
  - set firewall group network-group LAB-NODES network '${agent_subnet}'
  - set firewall ipv4 forward filter rule 40 description 'lab -> internal DNS (udp)'
  - set firewall ipv4 forward filter rule 40 source group network-group 'LAB-NODES'
  - set firewall ipv4 forward filter rule 40 destination address '${dns_ip}'
  - set firewall ipv4 forward filter rule 40 destination port '53'
  - set firewall ipv4 forward filter rule 40 protocol 'udp'
  - set firewall ipv4 forward filter rule 40 action 'accept'
  - set firewall ipv4 forward filter rule 41 description 'lab -> internal DNS (tcp)'
  - set firewall ipv4 forward filter rule 41 source group network-group 'LAB-NODES'
  - set firewall ipv4 forward filter rule 41 destination address '${dns_ip}'
  - set firewall ipv4 forward filter rule 41 destination port '53'
  - set firewall ipv4 forward filter rule 41 protocol 'tcp'
  - set firewall ipv4 forward filter rule 41 action 'accept'
```

(Rule 32 gives control full reach to the appliance for Ansible/SSH management on the mgmt VLAN; rules 40/41 are the query path. The DNS node's own egress for DoT + install is already covered by rule 20 `LAN -> internet`.)

## Task 4: VyOS DHCP option 6 for the node VLANs

**Files:**
- Modify: `router.tf` (add `dhcp_subnets` local + template var)
- Modify: `cloud-init/vyos-router.yaml.tftpl` (DHCP server block)

**Interfaces:**
- Consumes: `local.subnets`, `local.node_ip["dns"]` (as `dns_ip`, already passed in Task 3).
- Produces: template var `dhcp_subnets` (list of `{name, subnet, subnet_id, router, range_start, range_end}`); a VyOS `service dhcp-server` block setting `option name-server = dns_ip` per node VLAN.

- [ ] **Step 1: Add the `dhcp_subnets` local and pass it to the template**

In `router.tf`, add this to the `locals { }` block above the `templatefile(` call (i.e. before `vyos_user_data = templatefile(...)`), or as a sibling local — put it just after the `locals {` opening line:

```hcl
  # DHCP scopes for the node VLANs. Static server nodes ignore DHCP; this hands
  # option 6 (name-server = internal resolver) to any other lab client that joins
  # a node VLAN. Range .100-.199 avoids the .1 gateway and the .10/.53 statics.
  dhcp_subnets = [
    for key in ["control", "services", "agent"] : {
      name        = key
      subnet      = "${var.network_prefix}.${local.subnets[key].octet}.0/${var.subnet_mask}"
      subnet_id   = local.subnets[key].vlan
      router      = "${var.network_prefix}.${local.subnets[key].octet}.1"
      range_start = "${var.network_prefix}.${local.subnets[key].octet}.100"
      range_end   = "${var.network_prefix}.${local.subnets[key].octet}.199"
    }
  ]
```

Then add to the `templatefile(...)` argument map (next to `dns_ip`):

```hcl
    dhcp_subnets    = local.dhcp_subnets
```

- [ ] **Step 2: Add the DHCP server block to the VyOS template**

In `cloud-init/vyos-router.yaml.tftpl`, append at the **end of the file** (after the firewall `%{ endif ~}` on line 107):

```yaml

  # ---- DHCP server: hand option 6 (internal DNS) to lab clients per node VLAN ----
%{ for d in dhcp_subnets ~}
  - set service dhcp-server shared-network-name ${d.name} subnet ${d.subnet} subnet-id ${d.subnet_id}
  - set service dhcp-server shared-network-name ${d.name} subnet ${d.subnet} option default-router '${d.router}'
  - set service dhcp-server shared-network-name ${d.name} subnet ${d.subnet} option name-server '${dns_ip}'
  - set service dhcp-server shared-network-name ${d.name} subnet ${d.subnet} range LAN start '${d.range_start}'
  - set service dhcp-server shared-network-name ${d.name} subnet ${d.subnet} range LAN stop '${d.range_end}'
%{ endfor ~}
```

(VyOS 1.4+ requires `subnet-id`; the `range` name is arbitrary — `LAN` here. `option name-server` is DHCP option 6.)
