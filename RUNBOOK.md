# v2e-tf — First-Run Runbook

A VyOS router fronting four VLAN subnets, with three nodes behind it, deployed by
**OpenTofu + cloud-init**. Everything is driven by `tofu apply` — there are no
manual console steps. Follow this top to bottom for a clean first deploy.

```
        your mac ── ssh -p 2201 v2e@<wan> ─┐ (DNAT → control:22)
                                           ▼
   vmbr0 (WAN) ── eth0 ─┐   VyOS router (created first)
   vmbr1 (trunk) ─ eth1 ┴─ vif 100/101/102/103
        VLAN 100  10.1.0.1/24  vyos-mgmt
        VLAN 101  10.1.1.1/24  control  → control  10.1.1.10  (Ubuntu)
        VLAN 102  10.1.2.1/24  services → services 10.1.2.10  (Ubuntu)
        VLAN 103  10.1.3.1/24  agent    → agent    10.1.3.10  (Debian)
```

> IP convention in this runbook: the `10.1.x.x` addresses are **generated by the
> VyOS config** and are identical on every deploy — they're shown literally.
> Anything else (`192.168.1.x`, `example.com`) is a **placeholder for your own
> host / LAN / domain** — swap in your real values.

**What you end up with:** the router boots and self-configures, waits
`router_boot_wait` (120s), then the three nodes come up. `control` is the hub of
two SSH trust meshes (`v2e` admin + `ansible` automation) and, on first boot,
clones the Ansible repo and runs the playbook across the mesh.

---

## Step 0 — One-time prerequisites (on the Proxmox host)

Do these once. They are *not* managed by Terraform.

1. **Templates exist:** VyOS `9000`, Ubuntu `9001`, Debian `9002` (the VMIDs are
   configurable — see `vyos_template_id` / `ubuntu_template_id` / `debian_template_id`).
   - VyOS **must** be a cloud-init-*built* qcow2 — build one with
     [vyos-build](https://github.com/vyos/vyos-build). The free nightly ISO has no
     cloud-init and will silently ignore your config.
   - Ubuntu/Debian: the upstream `genericcloud` qcow2 images work as-is.

2. **Enable the `snippets` content type** on the `local` datastore (cloud-init
   user-data is uploaded there via SSH):
   ```bash
   pvesm set local --content iso,vztmpl,backup,snippets
   ```

3. **Make `vmbr1` VLAN-aware** (the trunk that carries the tagged VLANs).
   UI: *Datacenter → node → Network → vmbr1 → Edit → tick "VLAN aware" → Apply*.

4. **Create an API token** and confirm **passwordless SSH** to the host. The
   provider uploads snippets over SSH (the API can't), so this must work
   *non-interactively from your mac*:
   ```bash
   ssh root@192.168.1.10 true      # your PVE host — must return with no prompt
   ssh-add -l                      # your key must be loaded in the agent (provider uses agent=true)
   ```

## Step 2 — Configure `terraform.tfvars`

`terraform.tfvars` is gitignored (it holds secrets). Three self-contained blocks:
**(A)** the base config (always), **(B)** the Cloudflare tunnel (optional), and in
Step 3 **(C)** state encryption (optional). Each tfvars block re-prints the file
so you can see exactly what's set so far.

### Block A — base config (required)

Edit the values at the top. SSH is **key-only** on every node; the one password
you set, `sudo_password`, is the `v2e` user's **sudo** password (not an SSH
login). The `ansible` automation account has **no password** (locked; reached
only internally via `sudo su ansible`). The router console password is
auto-generated then **hashed** (VyOS needs a sha-512 hash, not plaintext).

```bash
# Make sure you have an SSH key (creates one if missing)
[ -f ~/.ssh/id_ed25519.pub ] || ssh-keygen -t ed25519

# ---- EDIT THESE ----
PVE_IP=192.168.1.10
PVE_TOKEN='terraform@pve!iac=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
WAN_CIDR='192.168.1.50/24'     # a FREE static IP on your LAN for the VyOS WAN
WAN_GW='192.168.1.1'           # your LAN gateway (VyOS's route to the internet)

# v2e sudo password (REQUIRED): a strong one is generated; replace with your own if you prefer.
SUDO_PW=$(openssl rand -base64 12)
echo "v2e sudo password (SAVE THIS): $SUDO_PW"

# Optional router console password: auto-generate, then hash with sha-512.
# (Key access already works without it; this is break-glass only.)
ROUTER_PW=$(openssl rand -base64 12)
ROUTER_HASH=$(openssl passwd -6 "$ROUTER_PW")
echo "router console password (SAVE THIS): $ROUTER_PW"

cat > terraform.tfvars <<EOF
proxmox_endpoint       = "https://${PVE_IP}:8006"
proxmox_api_token      = "${PVE_TOKEN}"
node_name              = "pve"
workstation_public_key = "$(cat ~/.ssh/id_ed25519.pub)"

wan_address = "${WAN_CIDR}"
wan_gateway = "${WAN_GW}"

sudo_password        = "${SUDO_PW}"      # v2e sudo password (SSH is key-only; required)
router_password_hash = "${ROUTER_HASH}"  # VyOS bootstrap 'vyos' user (optional)
EOF

chmod 600 terraform.tfvars
cat terraform.tfvars                                  # review what's set
```

> Password mapping: `sudo_password` = the `v2e` user's sudo password (**required**;
> SSH itself is key-only) · `router_password_hash` = the VyOS bootstrap `vyos`
> user (optional). The `ansible` account has **no** password (locked). There is
> **no** `cluster_password` / `ansible_password` / `agent_password` /
> `router_password` — `tofu` rejects unknown names.

### Block B — Cloudflare SSH tunnel (optional)

A key-auth SSH path to `control` through a Cloudflare tunnel (cloudflared is
installed on control at first boot). Skip this block entirely and the lab deploys
unchanged. Requires the domain already on Cloudflare; the API token needs scopes
*Account → Cloudflare Tunnel:Edit*, *Zone → DNS:Edit*, *Zone → Zone:Read*. This
**appends** to the tfvars from Block A.

> Clean teardown: the tunnel, its ingress config, and the proxied CNAME are all
> Terraform-managed, so `tofu destroy` **removes them from your Cloudflare
> account** automatically — no orphaned tunnel or stray `lab.example.com` DNS
> record left behind.

```bash
# ---- EDIT THESE ----
CF_API_TOKEN='cfut_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_ACCOUNT_ID='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_ZONE_ID='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_DOMAIN='example.com'        # a zone already on your Cloudflare account
CF_SUBDOMAIN='lab'             # public host becomes lab.example.com

cat >> terraform.tfvars <<EOF

cloudflare_api_token  = "${CF_API_TOKEN}"
cloudflare_account_id = "${CF_ACCOUNT_ID}"
cloudflare_zone_id    = "${CF_ZONE_ID}"
tunnel_hostname       = "${CF_SUBDOMAIN}.${CF_DOMAIN}"
tunnel_dns_name       = "${CF_SUBDOMAIN}"
EOF

chmod 600 terraform.tfvars
cat terraform.tfvars                                  # review: base + Cloudflare
```

## Step 3 — Encrypt the OpenTofu state (optional)

State is **plaintext by default** and holds secrets (both mesh SSH private keys,
the Proxmox token, the Cloudflare token, the node passwords). Encryption is
**opt-in via the `TF_ENCRYPTION` env var** — nothing lives in the repo. Skip it
and `tofu` just prints a harmless WARNING each run that state is plaintext.

> ⚠️ Once you migrate on-disk state to encrypted, `TF_ENCRYPTION` must be set for
> **every** tofu command. Lose the passphrase → state is **unrecoverable**.

### Block C — stash a passphrase + enable encryption for this shell

The `2>/dev/null` makes the Keychain write idempotent (ignores "already exists"
on re-runs). Back up the printed passphrase, then re-run only the `export` part
in any new shell before using `tofu`.

```bash
security add-generic-password -a "$USER" -s tofu-v2e-state -w "$(openssl rand -base64 32)" 2>/dev/null
echo "BACK UP THIS PASSPHRASE: $(security find-generic-password -a "$USER" -s tofu-v2e-state -w)"

export TF_ENCRYPTION="$(cat <<EOF
key_provider "pbkdf2" "main" { passphrase = "$(security find-generic-password -a "$USER" -s tofu-v2e-state -w)" }
method "aes_gcm" "main" { keys = key_provider.pbkdf2.main }
state { method = method.aes_gcm.main }
plan  { method = method.aes_gcm.main }
EOF
)"
```

> Brand-new deploy = no state yet, so just run Block C and continue. Converting an
> *existing* plaintext state? See the migration appendix at the bottom.

## Step 4 — Deploy

```bash
tofu init        # first time only — downloads providers (lock file is committed)
tofu plan        # review: 1 keypair set + router (+snippet) + 3 nodes (+snippets) + time_sleep
tofu apply       # type 'yes'
```

`apply` creates the router first, waits `router_boot_wait` (120s) for VyOS to
boot and apply routing, then builds the three nodes.

## Step 5 — Connect

VMs are recreated on each apply, so their host keys change. Clear stale ones
first, then connect.

```bash
WAN_IP=192.168.1.50             # your VyOS WAN address (host part of wan_address)
ssh-keygen -R "$WAN_IP" ; ssh-keygen -R "[$WAN_IP]:2201"

ssh vyos@"$WAN_IP"              # router, by key (bootstrap 'vyos' user; Ansible takes over later)
ssh -p 2201 v2e@"$WAN_IP"       # control node, by key (via VyOS DNAT WAN:2201 → control:22)
```

> From outside (the WAN port-forward or the Cloudflare tunnel) only **`v2e`** can
> log in, and by key only — `root` login is disabled and `ansible` is reachable
> only from inside (`sudo -iu ansible`, or control's mesh).

From **control**, both trust meshes work by key (private keys live only on control):

```bash
# v2e (admin) mesh — hub = control:
v2e@control:~$ ssh services
v2e@control:~$ ssh agent
v2e@control:~$ ssh vyos          # manage the router (=> vyos@10.1.1.1)

# ansible (automation) mesh — hub = control, NOPASSWD sudo everywhere:
v2e@control:~$ sudo -iu ansible
ansible@control:~$ ssh services ; ssh agent ; ssh vyos
```

### Via the Cloudflare tunnel (if you enabled Block B)

One-time setup on your mac, then SSH straight to the public hostname — no WAN
port-forward needed:

```bash
brew install cloudflared

CF_HOSTNAME=lab.example.com
cat >> ~/.ssh/config <<EOF

Host ${CF_HOSTNAME}
  ProxyCommand cloudflared access ssh --hostname %h
  User v2e
EOF

ssh "${CF_HOSTNAME}"            # => v2e@control through the tunnel (key auth)
```

`tofu output` prints all connection hints (WAN, mesh keys, tunnel status):

```bash
tofu output
tofu output ssh_to_control_via_tunnel
```

## Day-2 operations

```bash
tofu apply -replace='proxmox_virtual_environment_vm.vyos'                  # rebuild just the router
tofu apply -replace='proxmox_virtual_environment_vm.node["agent"]'        # rebuild one node
tofu destroy                                                              # tear it all down
```

> `tofu destroy` also deletes the Cloudflare tunnel + its proxied CNAME from your
> account (they're Terraform-managed) — teardown leaves nothing dangling there.

> cloud-init runs **once** per VM. Changing `terraform.tfvars` does **not**
> re-apply on a live VM — use `-replace` (above) to recreate the affected VM.

## Troubleshooting & hard-won gotchas (don't re-learn these)

- **`Unsupported argument` on apply** → a tfvars key that isn't a real variable.
  The password vars are exactly `sudo_password` and `router_password_hash` (not
  `cluster_password` / `ansible_password` / `agent_password` / `router_password`).
- **`sudo_password` is required** → omitting it, or using a known-weak value
  (`v2e` / `ansible` / `password` / `changeme`), fails validation.
- **Router password must be a sha-512 hash.** `var.router_password_hash` is fed
  straight to VyOS, so plaintext becomes an unusable console password. Hash with
  **OpenSSL 3.x** `openssl passwd -6` (Homebrew: `brew install openssl`). The
  macOS *system* `openssl` (LibreSSL) has no `-6`, and python's `crypt` silently
  falls back to DES — both produce junk. Leaving it empty = key-only access.
- **Nodes' apt step fails on first boot** → VyOS wasn't routing yet. Bump
  `router_boot_wait` (it's a fixed delay, not a health check). SSH/users/keys are
  set before packages, so login still works even if apt failed.
- **VyOS needs a cloud-init-*built* image.** The free nightly ISO has no
  cloud-init; `install image` templates silently ignore your config. Build the
  qcow2 with [vyos-build](https://github.com/vyos/vyos-build).
- **Router login (keys/password) goes in TOP-LEVEL cloud-config**, not
  `vyos_config_commands` — VyOS cloud-init never runs a `commit`.
- **`wan_gateway` is whatever NATs the WAN bridge to the internet** (your LAN
  gateway / PVE host) — not a separate router.
- **`tofu apply` SSHes to the PVE node** to upload snippets (not the API URL).
  Wrong network / key not in agent = "unreachable" failures.
- **Host keys change on every recreate** → `ssh-keygen -R` before reconnecting
  (Step 5).
- **Pasting into zsh:** the `PVE_TOKEN='…!…'` single quotes keep the `!` from
  triggering history expansion; the heredocs here are paste-safe. Don't paste
  stray `# comments` or stacked `ssh` lines.
- **The cluster private keys are embedded** in control's cloud-init drive
  (unavoidable for this pattern) — treat that VM's disk/snapshots accordingly.

## Appendix — Migrating an existing plaintext state to encrypted

Only needed once, if you already deployed without encryption and now want it on.
Adds a temporary `unencrypted` fallback so the first read loads the current
plaintext, then rewrites the state encrypted.

```bash
cp terraform.tfstate terraform.tfstate.preencrypt.bak     # safety copy
export TF_ENCRYPTION="$(cat <<EOF
key_provider "pbkdf2" "main" { passphrase = "$(security find-generic-password -a "$USER" -s tofu-v2e-state -w)" }
method "aes_gcm" "main" { keys = key_provider.pbkdf2.main }
method "unencrypted" "migrate" {}
state {
  method = method.aes_gcm.main
  fallback { method = method.unencrypted.migrate }
}
plan {
  method = method.aes_gcm.main
  fallback { method = method.unencrypted.migrate }
}
EOF
)"
tofu apply                                   # rewrites state ENCRYPTED
head -c 120 terraform.tfstate                # confirm: encryption metadata, NOT readable attrs

# Re-export the NORMAL form (Block C, no 'migrate' method / no fallback), then:
tofu plan                                    # confirms it reads encrypted, zero drift
rm -P terraform.tfstate.backup terraform.tfstate.preencrypt.bak   # shred plaintext leftovers
```

Gotchas:
- `TF_ENCRYPTION` must be set for **every** command after migration (`init` included).
- **Don't rename** the `pbkdf2 "main"` / `aes_gcm "main"` labels — ciphertext stores metadata keyed to them.
- `terraform.tfstate.backup` stays **plaintext** after migration → shred it (above).
