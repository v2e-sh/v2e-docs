# v2e — First-Run Runbook (full stack)

Deploy the whole lab from a bare Proxmox host to **apps served over HTTPS**, in one
pass. A VyOS router fronts four VLAN subnets with three nodes behind it; OpenTofu +
cloud-init build the infrastructure, then control's first boot runs Ansible, which
hardens the nodes, installs Docker, and brings up the Traefik + TinyAuth + whoami
compose stack.

```
        your mac ── ssh -p 2201 v2e@<wan> ─┐ (DNAT → control:22)
                                           ▼
   vmbr0 (WAN) ── eth0 ─┐   VyOS router (created first)
   vmbr1 (trunk) ─ eth1 ┴─ vif 100/101/102/103
        VLAN 100  10.1.0.1/24  vyos-mgmt
        VLAN 101  10.1.1.1/24  control  → control  10.1.1.10  (Ubuntu, mesh hub)
        VLAN 102  10.1.2.1/24  services → services 10.1.2.10  (Ubuntu, Docker/Traefik)
        VLAN 103  10.1.3.1/24  agent    → agent    10.1.3.10  (Debian)
```

> **IP convention:** the `10.1.x.x` addresses are generated by the VyOS config and are
> identical on every deploy — shown literally. Anything else (`192.168.1.x`,
> `example.com`, `v2e.sh`) is a **placeholder** — swap in your real host / LAN / domain.

---

## The whole deploy at a glance

Nine steps. Steps 1–5 are on your mac; the rest is automatic + verification.

```
PREP   0. Prerequisites: PVE host, Cloudflare domain + 2 tokens, build templates
CONFIG 1. git clone v2e-tf
       2. terraform.tfvars — Block A (base)
       3. Author secrets  — Block C (Cloudflare DNS token + TinyAuth user)   ← REQUIRED for apps
       4. Set your domain (only if it isn't v2e.sh)
DEPLOY 5. tofu init && tofu apply         → router + 3 nodes; control auto-runs Ansible
VERIFY 6. Watch it converge: cloud-init → Ansible PLAY RECAP → docker → cert
       7. Flip to production certs, re-run
       8. Reach the apps (from control today; browser via DNS-1/Tailscale later)
```

**End state:** `https://whoami.v2e.sh` returns JSON with a valid Let's Encrypt cert, and
`https://traefik.v2e.sh` prompts for TinyAuth login — reachable from the control node.

> **New to this repo? Two things will bite you if skipped:** (1) Step 3 is **mandatory**
> for the app stack — without the Cloudflare DNS token + a TinyAuth user, Ansible stops at
> the deploy phase. (2) Templates build to staging IDs `9900–9902`; Terraform defaults to
> `9000–9002` — align them (Step 0 / Step 2) or the first `apply` clones a VM that doesn't
> exist.

# Part 1 — Prerequisites (Step 0)

Do these once. They are **not** managed by Terraform.

### Your mac

```bash
brew install opentofu git sops age    # sops + age are needed for Step 3
ssh-add -l                            # your SSH key must be loaded (the provider uses agent=true)
[ -f ~/.ssh/id_ed25519.pub ] || ssh-keygen -t ed25519
```

### Proxmox host

1. **Enable the `snippets` content type** on `local` (cloud-init user-data is uploaded there over SSH):
   ```bash
   pvesm set local --content iso,vztmpl,backup,snippets
   ```
2. **Make `vmbr1` VLAN-aware** (the trunk carrying the tagged VLANs):
   *Datacenter → node → Network → vmbr1 → Edit → tick "VLAN aware" → Apply.*
3. **API token + passwordless SSH.** The provider uploads snippets over SSH (the API
   can't), so this must work non-interactively **from your mac**:
   ```bash
   ssh root@192.168.1.10 true      # your PVE host — must return with no prompt
   ```

### Cloudflare — domain + **two** tokens

You need a domain already on Cloudflare. The stack uses **two different scoped tokens**
(don't confuse them):

| Token | Scopes | Used for | Required? |
|---|---|---|---|
| **DNS-01 token** | Zone:Read + DNS:Edit | Traefik wildcard cert (Step 3) | **Yes** (for HTTPS) |
| **Tunnel token** | Account→Cloudflare Tunnel:Edit, Zone→DNS:Edit, Zone→Zone:Read | SSH-over-tunnel (Appendix A) | No (optional) |

Create the DNS-01 token at *Cloudflare → My Profile → API Tokens → Create → Custom* and
save it for Step 3.

### Build the VM templates

Templates are built by the **`v2e-templates`** repo, **on the PVE host** (`virt-customize` +
`qm`), each image customized offline → imported → turned into a `qm template`.

> **Prereq (PVE host):** `apt install libguestfs-tools`, cloud-init-capable storage
> (`local-lvm`), internet from the host.

```bash
# On the PVE host:
git clone https://github.com/v2e-sh/v2e-templates.git && cd v2e-templates
$EDITOR config.env          # STORAGE, BRIDGE, per-image *_VMID, image *_URL/*_SHA256
make ubuntu debian          # cloud images, ~2 min each   (Parrot optional: make parrot)
```

VyOS has no upstream cloud image — pick one path:

```bash
make vyos                   # A) fetch the pinned qcow2 (VYOS_URL/VYOS_SHA256 preset) — fast
make vyos-build && make vyos # B) build rolling from source in a throwaway Debian VM (~20-40 min)
```

> Path B clones the **Debian** template first and needs host-NAT internet — set
> `VYOS_BUILDER_IP` / `BUILD_GW` in `config.env`. Nodes get `sops` + `age` baked in; **VyOS
> gets neither** (it doesn't run Ansible).

**⚠️ VMID staging → production.** Builds land at staging `9900–9903`; Terraform defaults to
`9000–9002`. Choose one:
- **Easiest:** keep the staging IDs and set them in `terraform.tfvars` (Step 2):
  `vyos_template_id = 9900`, `ubuntu_template_id = 9901`, `debian_template_id = 9902`.
- **Or promote:** edit the `*_VMID` in `config.env` to the prod IDs and re-run `make`
  (a CLI `VMID=…` is ignored — each `build-*.sh` re-sources `config.env`).

Verify a template before relying on it:
```bash
qm clone 9901 9950 --name verify
qm set 9950 --ipconfig0 ip=10.0.0.50/24,gw=10.0.0.1 --ciuser test --cipassword test
qm start 9950 && qm guest exec 9950 -- cloud-init status   # done; sops + age present
qm stop 9950 && qm destroy 9950 --purge
```

# Part 2 — Configure & deploy

## Step 1 — Clone the project

```bash
git clone https://github.com/v2e-sh/v2e-tf.git
cd v2e-tf
```

## Step 2 — `terraform.tfvars` (Block A, required)

`terraform.tfvars` is gitignored (it holds secrets). SSH is **key-only** on every node.
The one password you set, `sudo_password`, is the `v2e` user's **sudo** password (not an
SSH login). The `ansible` automation account is passwordless/locked. The router console
password is optional (break-glass) and must be a **sha-512 hash**.

```bash
# ---- EDIT THESE ----
PVE_IP=192.168.1.10
PVE_TOKEN='terraform@pve!iac=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
WAN_CIDR='192.168.1.50/24'     # a FREE static IP on your LAN for the VyOS WAN
WAN_GW='192.168.1.1'           # your LAN gateway (VyOS's route to the internet)

SUDO_PW=$(openssl rand -base64 12)
echo "v2e sudo password (SAVE THIS): $SUDO_PW"
ROUTER_PW=$(openssl rand -base64 12)                 # optional console break-glass
ROUTER_HASH=$(openssl passwd -6 "$ROUTER_PW")        # sha-512; needs OpenSSL 3.x (brew), not macOS LibreSSL
echo "router console password (SAVE THIS): $ROUTER_PW"

cat > terraform.tfvars <<EOF
proxmox_endpoint       = "https://${PVE_IP}:8006"
proxmox_api_token      = "${PVE_TOKEN}"
node_name              = "pve"
workstation_public_key = "$(cat ~/.ssh/id_ed25519.pub)"

wan_address = "${WAN_CIDR}"
wan_gateway = "${WAN_GW}"

sudo_password        = "${SUDO_PW}"      # v2e user's sudo password (SSH is key-only; required)
router_password_hash = "${ROUTER_HASH}"  # VyOS bootstrap 'vyos' user (optional)

# If you kept the staging template IDs from Step 0, uncomment:
# vyos_template_id   = 9900
# ubuntu_template_id = 9901
# debian_template_id = 9902

# Pin Ansible so a breaking upstream release can't derail first boot (recommended):
# ansible_version = "11.1.0"
EOF

chmod 600 terraform.tfvars
cat terraform.tfvars                                 # review
```

> Only `sudo_password` and `router_password_hash` are password vars — `tofu` rejects any
> other name (`cluster_password`, `ansible_password`, …). `sudo_password` is required and
> rejects known-weak values.

## Step 3 — Author the deployment secrets (REQUIRED for the app stack)

Ship one **age-encrypted** file to control. Ansible reads it and renders the compose
stack's `.env`, so **without it the services phase stops** (`compose_stack` asserts the two
keys are present). At first boot control writes the age key to `~/.config/sops/age/keys.txt`
and the file to `~/ansible/group_vars/all.sops.yaml`.

> **The keys must be lowercase** — `cf_dns_api_token` and `tinyauth_auth_users`. Ansible's
> `env.j2` reads those names and maps them to the `CF_DNS_API_TOKEN` / `TINYAUTH_AUTH_USERS`
> env vars the compose files expect. (The uppercase keys in
> `v2e-compose/secrets.sops.yaml.example` are for the *local* `make up` dev workflow only —
> a different path. Use lowercase here.)

**a. Create a TinyAuth user** (produces `user:bcrypt`):
```bash
docker run --rm -it ghcr.io/steveiliop56/tinyauth:v5.0.7 user create
# copy the resulting  user:$2a$10$...  string
```

**b. Author + encrypt the file** (needs the Cloudflare **DNS-01** token from Step 0). Edit
the two values at the top, then paste the rest as-is:
```bash
# ---- EDIT THESE ----  (single quotes keep the bcrypt '$' chars literal)
CF_DNS_TOKEN='cloudflare-DNS-01-token'   # scoped Zone:Read + DNS:Edit
TINYAUTH_USER='user:$2a$10$...'          # the string from step (a)

# 1. Author the plaintext secrets (lowercase keys — required):
cat > secrets.yaml <<EOF
cf_dns_api_token: "${CF_DNS_TOKEN}"
tinyauth_auth_users: "${TINYAUTH_USER}"
EOF

# 2. Key control decrypts with (this one ships to control):
age-keygen -o keys.txt
AGE_PUB=$(grep -oE 'age1[0-9a-z]+' keys.txt | head -1)

# 3. Backup/escrow key — store keys-backup.txt OFFLINE. Under SOPS a lost private key
#    loses EVERY secret; a second recipient makes a lost keys.txt recoverable.
age-keygen -o keys-backup.txt
AGE_BACKUP=$(grep -oE 'age1[0-9a-z]+' keys-backup.txt | head -1)

# 4. Encrypt to BOTH recipients, then shred the plaintext:
sops --encrypt --age "${AGE_PUB},${AGE_BACKUP}" secrets.yaml > secrets.sops.yaml
rm -P secrets.yaml

# 5. Point tfvars at them:
cat >> terraform.tfvars <<EOF

sops_secrets_file = "./secrets.sops.yaml"   # -> control:~/ansible/group_vars/all.sops.yaml
sops_age_key_file = "./keys.txt"            # -> control:~/.config/sops/age/keys.txt
EOF
chmod 600 terraform.tfvars keys.txt keys-backup.txt secrets.sops.yaml
cat terraform.tfvars                                 # review: base + SOPS
```

> Set **both** `sops_*` vars or **neither** (a half-config fails a `tofu` check). Keep
> `keys.txt` / `keys-backup.txt` out of git and the backup offline. Edit secrets later with
> `sops secrets.sops.yaml`. The age private key is read into tf state — enable state
> encryption (Appendix B) and rotate on a clean rebuild.

## Step 4 — Set your domain (only if it isn't `v2e.sh`)

The compose stack's domain, ACME email, and cert resolver live in **v2e-ansible**'s
`inventory/group_vars/services.yml` (not in tfvars — group_vars win over the SOPS file). The
default is `v2e.sh`. If that's your domain, skip this step.

Otherwise fork `v2e-ansible`, edit those values, and point control at your fork (set
`ansible_repo_url` in `terraform.tfvars`). The domain **must be the Cloudflare zone your
DNS-01 token is scoped to**, or the wildcard cert won't validate.

```yaml
# v2e-ansible/inventory/group_vars/services.yml
compose_stack_domain: "example.com"
compose_stack_acme_email: "admin@example.com"
compose_stack_cert_resolver: staging        # leave staging for the first run (Step 7 flips it)
```

## Step 5 — Deploy

```bash
tofu init        # first time only — downloads providers (lock file is committed)
tofu plan        # review: keypairs + router (+snippet) + 3 nodes (+snippets) + time_sleep
tofu apply       # type 'yes'
```

`apply` creates the router first, waits `router_boot_wait` (120s) for VyOS to boot and route,
then builds the three nodes. Control's cloud-init then installs Ansible, clones
`v2e-ansible`, installs its Galaxy roles, and runs `site.yml` across the mesh — all
automatically. Give it several minutes after `apply` returns.

> State is plaintext by default and holds secrets (mesh keys, tokens, the age key, node
> passwords). Encrypting it is **recommended** — see **Appendix B** (do it before the first
> apply).

# Part 3 — Verify the full stack

## Step 6 — Watch it converge

First connect to control (host keys change on every rebuild, so clear stale ones):

```bash
WAN_IP=192.168.1.50                                   # host part of wan_address
ssh-keygen -R "[$WAN_IP]:2201"
ssh -p 2201 v2e@"$WAN_IP"                             # control, by key (WAN:2201 → control:22)
```

Then verify each layer in order — **stop at the first one that fails** and see Appendix E.
Set your domain once (matches Step 4; the checks below reuse it):

```bash
# ---- EDIT THIS ----
v2e@control:~$ DOMAIN=v2e.sh                           # your compose_stack_domain

# 1. Cloud-init finished on control
v2e@control:~$ sudo cloud-init status --long          # status: done

# 2. Ansible ran all three phases (bootstrap → services → applications)
v2e@control:~$ sudo grep -E "PLAY RECAP|failed=" /var/log/cloud-init-output.log | tail -20
#    every host should show  failed=0

# 3. SOPS secrets actually decrypted (not raw ENC[...] ciphertext)
v2e@control:~$ sudo -iu ansible sops -d ~/ansible/group_vars/all.sops.yaml
#    prints your plaintext cf_dns_api_token / tinyauth_auth_users

# 4. Docker + the stack are healthy on services
v2e@control:~$ ssh services 'docker ps --format "{{.Names}}\t{{.Status}}"'
#    traefik, tinyauth, whoami all Up

# 5. Traefik issued a cert and whoami answers over HTTPS (staging = self-signed, so -k).
#    Double-quoted so control expands $DOMAIN before running curl on services.
v2e@control:~$ ssh services "curl -sk --resolve whoami.$DOMAIN:443:10.1.2.10 https://whoami.$DOMAIN | head"
#    whoami JSON (Hostname, IP, …)
v2e@control:~$ ssh services "curl -skI --resolve whoami.$DOMAIN:443:10.1.2.10 https://whoami.$DOMAIN | grep -i server"
```

> **Why staging first / why `-k`.** `cert_resolver` defaults to `staging`, so the first
> cert is from Let's Encrypt's **staging** CA (untrusted — browsers/`curl` warn). This is
> deliberate: staging has generous rate limits, so you validate the DNS-01 flow without
> risking a production-CA lockout. Step 7 switches to a real cert.

## Step 7 — Go to production certs

Once the staging cert issues cleanly, flip to the production resolver and re-run Ansible:

```bash
# On your mac: v2e-ansible/inventory/group_vars/services.yml
compose_stack_cert_resolver: production
```
```bash
# Apply it (re-run the playbook from control):
v2e@control:~$ sudo -iu ansible bash -lc 'cd ~/ansible && git pull && ansible-playbook -i inventory/hosts.ini site.yml'
# then re-check — now WITHOUT -k (should be a trusted cert; $DOMAIN from Step 6, or re-set it):
v2e@control:~$ ssh services "curl -s --resolve whoami.$DOMAIN:443:10.1.2.10 https://whoami.$DOMAIN | head"
```

> Production and staging use separate storage (`acme.json` vs `acme-staging.json`), so the
> flip re-issues cleanly. If you forked, `git pull` on control tracks your fork's `main`.

## Step 8 — Reach the apps

Today the only WAN entry point is `control:22` (the SSH DNAT). Traefik listens on
`services:443`, which has **no public route yet** — browser access by name is a planned
feature (DNS-1 internal resolver + Tailscale). So:

- **From control (works now):** the `curl --resolve` checks in Step 6.
- **In your browser now (interim):** forward services:443 over SSH and fake the DNS locally:
  ```bash
  # on your mac:
  ssh -p 2201 -L 8443:10.1.2.10:443 v2e@"$WAN_IP"
  # add to /etc/hosts:   127.0.0.1  whoami.v2e.sh traefik.v2e.sh tinyauth.v2e.sh
  # then browse https://whoami.v2e.sh:8443  (accept the staging-cert warning until Step 7)
  ```
- **Later (the real path):** DNS-1 publishes `*.int.v2e.sh → 10.1.2.10` and Tailscale routes
  it, so `https://traefik.v2e.sh` opens directly. See the DNS-1 phase in the master plan.

`https://traefik.<domain>` and `https://tinyauth.<domain>` are gated by TinyAuth (log in
with the user from Step 3a); `whoami` is open.

# Appendix A — Cloudflare SSH tunnel (optional)

A key-auth SSH path to control through a Cloudflare tunnel (no WAN port-forward). Requires
the domain on Cloudflare and the **tunnel** token (Step 0 table). Append to your tfvars:

```bash
# ---- EDIT THESE ----
CF_API_TOKEN='cfut_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_ACCOUNT_ID='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_ZONE_ID='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_DOMAIN='example.com'
CF_SUBDOMAIN='lab'             # public host becomes lab.example.com
CLOUDFLARE_ACCESS_EMAIL=""     # optional OTP gate; empty = SSH key alone

cat >> terraform.tfvars <<EOF

cloudflare_api_token  = "${CF_API_TOKEN}"
cloudflare_account_id = "${CF_ACCOUNT_ID}"
cloudflare_zone_id    = "${CF_ZONE_ID}"
tunnel_hostname       = "${CF_SUBDOMAIN}.${CF_DOMAIN}"
tunnel_dns_name       = "${CF_SUBDOMAIN}"
EOF
[ -n "$CLOUDFLARE_ACCESS_EMAIL" ] && echo "cloudflare_access_emails = [\"$CLOUDFLARE_ACCESS_EMAIL\"]" >> terraform.tfvars
chmod 600 terraform.tfvars
```

Then, on your mac:
```bash
brew install cloudflared
CF_HOSTNAME=lab.example.com
cat >> ~/.ssh/config <<EOF

Host ${CF_HOSTNAME}
  ProxyCommand cloudflared access ssh --hostname %h
  User v2e
EOF
ssh "${CF_HOSTNAME}"           # => v2e@control through the tunnel (key auth)
```

> `tofu destroy` removes the tunnel + its proxied CNAME from your Cloudflare account — no
> orphans. `tofu output ssh_to_control_via_tunnel` prints the hint.

# Appendix B — Encrypt the OpenTofu state (recommended)

State is plaintext by default and holds every secret. Encryption is opt-in via the
`TF_ENCRYPTION` env var (nothing lives in the repo). Do it **before** the first apply.

```bash
security add-generic-password -a "$USER" -s tofu-v2e-state -w "$(openssl rand -base64 32)" 2>/dev/null
echo "BACK UP THIS PASSPHRASE: $(security find-generic-password -a "$USER" -s tofu-v2e-state -w)"

export TF_ENCRYPTION="$(cat <<EOF
key_provider "pbkdf2" "main" { passphrase = "$(security find-generic-password -a "$USER" -s tofu-v2e-state -w)" }
method "aes_gcm" "main" { keys = key_provider.pbkdf2.main }
state { method = method.aes_gcm.main }
plan  { method = method.aes_gcm.main }
EOF
)"
```

> Once migrated, `TF_ENCRYPTION` must be set for **every** tofu command (re-run the `export`
> in each new shell). Lose the passphrase → state is unrecoverable. Skipping this only costs
> a harmless plaintext-state WARNING each run.

**Migrating an existing plaintext state** (only if you already applied without encryption):
```bash
cp terraform.tfstate terraform.tfstate.preencrypt.bak
export TF_ENCRYPTION="$(cat <<EOF
key_provider "pbkdf2" "main" { passphrase = "$(security find-generic-password -a "$USER" -s tofu-v2e-state -w)" }
method "aes_gcm" "main" { keys = key_provider.pbkdf2.main }
method "unencrypted" "migrate" {}
state { method = method.aes_gcm.main  fallback { method = method.unencrypted.migrate } }
plan  { method = method.aes_gcm.main  fallback { method = method.unencrypted.migrate } }
EOF
)"
tofu apply                                   # rewrites state ENCRYPTED
# then re-export the NORMAL form above (no 'migrate'/fallback) and:
tofu plan                                    # reads encrypted, zero drift
rm -P terraform.tfstate.backup terraform.tfstate.preencrypt.bak
```
> Don't rename the `pbkdf2 "main"` / `aes_gcm "main"` labels — ciphertext stores metadata
> keyed to them. `terraform.tfstate.backup` stays plaintext after migration → shred it.

# Appendix C — Firewall (router isolation)

Default-deny, **defined in Terraform**, applied by cloud-init at router build.

- **WAN in:** only `control:22` (the DNAT). Router SSH over WAN only from
  `trusted_mgmt_sources` (default: none — reach it via control).
- **Inside:** `control` (hub) → `services` and `agent`; spokes can't reach each other or
  initiate back to control; `agent` is isolated. Return traffic for allowed flows is
  stateful.
- **Egress:** node → internet is open. (A deny-by-default agent allowlist exists on the
  `fix/tf-hardening-state` branch — not on `main` yet.)

Knobs in `terraform.tfvars`:
```
firewall_enabled     = true                 # false disables the whole firewall
trusted_mgmt_sources = ["203.0.113.5/32"]   # CIDRs allowed to SSH the router over the WAN
```
Rules live in `cloud-init/vyos-router.yaml.tftpl`. Cloud-init applies them **once** at
router build, so changes take effect on the next router rebuild:
```bash
tofu apply -replace='proxmox_virtual_environment_vm.vyos'   # keep the PVE console open in case a rule locks you out
```
> A live `configure`→`set firewall` edit works but is wiped on the next rebuild (and
> phase-2 Ansible later owns router config).

# Appendix D — Day-2 operations

```bash
tofu apply -replace='proxmox_virtual_environment_vm.vyos'                # rebuild just the router
tofu apply -replace='proxmox_virtual_environment_vm.node["agent"]'       # rebuild one node
tofu destroy                                                             # tear it all down
```
> cloud-init runs **once** per VM — changing tfvars does not re-apply on a live VM; use
> `-replace`. `tofu destroy` also deletes the Cloudflare tunnel + CNAME.
>
> Re-running Ansible without a rebuild: `sudo -iu ansible bash -lc 'cd ~/ansible && git pull
> && ansible-playbook -i inventory/hosts.ini site.yml'` from control. Operational playbooks
> are tag-gated: `--tags vyos` (router hardening), `--tags patch`. The agent kill-switch is
> a standalone playbook: `ansible-playbook playbooks/ops/killswitch.yml --tags cut` (and
> `allow` / `cut-hard`).

> **DR** (state backup, age-key rotation/escrow, snapshot restore) is the DOCS-2 runbook,
> not yet written. Highest-leverage piece: keep the Step 3 escrow key safe — under SOPS a
> lost private key loses every secret.

# Appendix E — Troubleshooting (don't re-learn these)

**Full-stack (Ansible + apps):**
- **Ansible phase failed / `PLAY RECAP` shows `failed>0`** → read the play name above it in
  `/var/log/cloud-init-output.log`. Common causes below.
- **`compose_stack needs cf_dns_api_token and tinyauth_auth_users`** → Step 3 was skipped or
  the keys are wrong. They must be **lowercase** in the SOPS file. Verify decryption:
  `sudo -iu ansible sops -d ~/ansible/group_vars/all.sops.yaml`.
- **Secrets show as `ENC[AES256_GCM,...]` instead of values** → the file wasn't
  auto-decrypted. It must land as `group_vars/all.sops.yaml` (the `.sops.yaml` extension is
  required by the community.sops vars plugin). Confirm with the `sops -d` command above; if
  that works but vars are still ciphertext, the file is named `all.yml` — rebuild control
  after updating v2e-tf.
- **Traefik cert never issues** → (staging) check the DNS-01 token scope (Zone:Read +
  DNS:Edit) and that your domain is the token's zone; `ssh services 'docker logs traefik
  2>&1 | grep -i acme'`. The DNS challenge uses `1.1.1.1` — ensure the services node has
  egress.
- **`docker ps` on services is empty** → the services phase didn't reach the deploy;
  re-check the PLAY RECAP and Galaxy-role install (`geerlingguy.docker`).
- **Can't reach the app from your mac** → expected; there's no WAN route to `services:443`
  yet. Use the from-control `curl --resolve` or the SSH-forward trick (Step 8).

**Infrastructure (Terraform + cloud-init):**
- **`Unsupported argument`** → a tfvars key that isn't a real variable (only `sudo_password`
  / `router_password_hash` for passwords).
- **`error cloning VM: unable to find configuration file for VM 9000`** → that template VMID
  doesn't exist. Build it or point `*_template_id` at what you have (staging `9900–9903` vs
  prod `9000–9003`) — see Step 0.
- **Router password must be a sha-512 hash** → use `openssl passwd -6` from **OpenSSL 3.x**
  (`brew install openssl`); macOS LibreSSL has no `-6`. Empty = key-only access.
- **Nodes' apt step fails on first boot** → VyOS wasn't routing yet. Bump `router_boot_wait`
  (a fixed delay, not a health check). SSH/keys are set before packages, so login still works.
- **VyOS needs a cloud-init-*built* image** — the free nightly ISO ignores your config.
- **`tofu apply` SSHes to the PVE node** to upload snippets (not the API URL). Wrong network
  / key not in agent = "unreachable".
- **Host keys change on every recreate** → `ssh-keygen -R` before reconnecting (Step 6).
- **`Still destroying… [id=3xx]`** → bpg waits on the guest agent; VMs set
  `stop_on_destroy=true` to avoid it. If one wedges: `qm stop <id> --skiplock && qm destroy
  <id> --purge --skiplock`, then `tofu state rm` it.
- **`plan`/`apply` stuck on `Refreshing… [id=3xx]`** → an unresponsive guest agent; use
  `tofu plan -refresh=false`.
- **First-boot apt upgrade** is on (`package_upgrade` default) so Ansible sees a patched
  system; if a desktop-class control OOMs during it, set `package_upgrade = false` and let
  Ansible patch.
- **SOPS "half-configured" check fails** → set **both** `sops_secrets_file` and
  `sops_age_key_file`, or neither.
- **Cloudflare `404 Tunnel not found`** → the tunnel in state was deleted out-of-band;
  `tofu state rm` the `cloudflare_*` resources and re-apply.
- **Pasting into zsh:** the `PVE_TOKEN='…!…'` single quotes stop `!` history expansion; the
  heredocs are paste-safe. Don't paste stray `# comments`.
