# v2e — First-Run Runbook

Deploy the whole lab from a bare Proxmox host to **apps served over HTTPS on an
internal domain**, in one pass. A VyOS router fronts four VLAN subnets with four
nodes behind it; **OpenTofu + cloud-init** build the infrastructure, then control's
first boot runs **Ansible**, which hardens the nodes, installs Docker, and brings up
the compose stacks.

Every value you must supply is set in **one of two files** you author in Step 2 and
Step 3 — `terraform.tfvars` and `secrets.sops.yaml`. The full variable reference
(every knob, what it does, how to generate it) is **[`CONFIGURATION.md`](CONFIGURATION.md)**;
this runbook only walks the happy path.

```
        your mac ── ssh -J root@<pve> -p 2201 v2e@<wan> ─┐ (DNAT → control:22)
                                                         ▼
   vmbr0 (WAN) ── eth0 ─┐   VyOS router 310 (created first)
   vmbr1 (trunk) ─ eth1 ┴─ vif 100/101/102/103
        VLAN 100  10.1.0.1/24  mgmt     → infra    10.1.0.10  (Debian: Technitium DNS, RustDesk relay)
        VLAN 101  10.1.1.1/24  control  → control  10.1.1.10  (ParrotOS desktop: mesh hub, Ansible)
        VLAN 102  10.1.2.1/24  services → services 10.1.2.10  (Ubuntu: Docker app estate)
        VLAN 103  10.1.3.1/24  agent    → agent    10.1.3.10  (Debian: AI agent, restricted egress)
```

> **IP convention:** the `10.1.x.x` addresses are generated by the VyOS config and are
> identical on every deploy — shown literally. Anything else (`192.168.1.x`,
> `example.com`) is a **placeholder** — swap in your real host / LAN / domain.
> Design rationale for each layer lives in `specs/` — referenced inline where relevant.

**End state:** `https://whoami.int.<your-domain>` answers with a production Let's
Encrypt wildcard cert, `https://grafana|semaphore|arcane|uptime|traefik.int.<domain>`
sit behind Authelia SSO login (single sign-on / OIDC) — all reachable from the control node (and from your mac
after the post-deploy access steps).

## The whole deploy at a glance

```
PREP    0. Prerequisites — mac tools, PVE host settings, Cloudflare domain + tokens
        1. Build the VM templates on the PVE host (v2e-templates)
CONFIG  2. terraform.tfvars — base config          (Block A, required)
        3. secrets.sops.yaml — app secrets          (Block B, required for the app stack)
        4. Optional: state encryption · Cloudflare SSH tunnel   (Blocks C, D)
DEPLOY  5. tofu init && tofu apply    → router + 4 nodes; control auto-runs Ansible
VERIFY  6. Watch it converge, then verify: cloud-init → PLAY RECAP → docker → certs
ACCESS  7. Reach the lab: SSH paths, browser access to the apps
MANUAL  8. Post-deploy manual bits (tailnet approval/split-DNS, RustDesk client pin)
```

## Step 0 — One-time prerequisites

### Your mac

```bash
brew install opentofu git sops age
ssh-add -l                            # your SSH key must be loaded (the provider uses agent=true)
[ -f ~/.ssh/id_ed25519.pub ] || ssh-keygen -t ed25519
```

> The Authelia admin credential is an argon2id hash (`authelia_admin_password_hash`),
> generated in Step 3 (see CONFIGURATION.md for the exact command); htpasswd/bcrypt no
> longer applies.

### Proxmox host

Do these once; they are *not* managed by Terraform.

1. **Enable the `snippets` content type** on `local` (cloud-init user-data is uploaded
   there over SSH):
   ```bash
   pvesm set local --content iso,vztmpl,backup,snippets
   ```
2. **Make `vmbr1` VLAN-aware** (the trunk carrying the tagged VLANs):
   *Datacenter → node → Network → vmbr1 → Edit → tick "VLAN aware" → Apply.*
3. **API token + passwordless SSH.** The provider uploads snippets over SSH (the API
   can't), so this must work non-interactively **from your mac**:
   ```bash
   ssh root@192.168.1.10 true      # your PVE host — must return with no prompt
   ```
   Create the token under *Datacenter → Permissions → API Tokens* (e.g.
   `terraform@pve!iac`), privilege separation off or with VM.* / Datastore.* / SDN
   privileges.

### Cloudflare — a domain + two scoped tokens

You need a domain already on Cloudflare. The stack uses **two different tokens** —
don't confuse them:

| Token | Scopes | Used for | Required? |
|---|---|---|---|
| **DNS-01 token** | Zone → Zone:Read + DNS:Edit (your zone only) | Traefik wildcard cert (Step 3, `cf_dns_api_token`) | **Yes** (for HTTPS) |
| **Tunnel token** | Account → Cloudflare Tunnel:Edit; Zone → DNS:Edit + Zone:Read | SSH-over-tunnel (Step 4, `cloudflare_api_token`) | No (optional) |

Create them at *Cloudflare → My Profile → API Tokens → Create Token → Custom*.

## Step 1 — Build the VM templates (`v2e-templates`)

Templates are built **on the PVE host** (`virt-customize` + `qm`): each image is
fetched (URL + SHA256 pinned), customized offline, imported, and turned into a
`qm template` — the artifact Terraform clones. Design:
`specs/2026-06-30-packer-templates-design.md`.

> **Prereq (PVE host):** `apt install libguestfs-tools`, cloud-init-capable storage
> (`local-lvm`), internet from the host. `make` is **not** installed on a stock
> Proxmox — call the build scripts directly.

```bash
# On the PVE host:
git clone https://github.com/v2e-sh/v2e-templates.git && cd v2e-templates
$EDITOR config.env          # STORAGE, BRIDGE, per-image *_VMID, image *_URL/*_SHA256

bash build-ubuntu.sh        # services            (~2 min)
bash build-debian.sh        # agent + infra       (~2 min)
bash build-parrot.sh        # control (desktop; UEFI-only image)
bash build-vyos.sh          # router — see below
```

**VyOS** has no upstream cloud image — either fetch the published qcow2
(`VYOS_URL`/`VYOS_SHA256` preset in `config.env`) with `bash build-vyos.sh`, or build
rolling from source in a throwaway Debian VM first (`bash vyos/build-from-source.sh`,
~20–40 min; needs `VYOS_BUILDER_IP`/`BUILD_GW` in `config.env` and the Debian template
built). The VyOS image **must** be cloud-init-built — the free nightly ISO has no
cloud-init and silently ignores your config.

**VMIDs — staging vs production.** Builds land at the staging VMIDs in `config.env`:

| Image | Staging VMID | Consumed by | tfvars variable |
|---|---|---|---|
| VyOS | 9900 | router | `vyos_template_id` |
| Ubuntu 24.04 | 9901 | services | `ubuntu_template_id` |
| Debian 13 | 9902 | agent, infra | `debian_template_id` |
| ParrotOS Home | 9903 | control | `parrot_template_id` |

Terraform defaults to prod `9000–9003`. Either point the four `*_template_id` tfvars
at the staging IDs (Step 2 does this), or edit `*_VMID` in `config.env` to the prod
IDs and rebuild. A CLI `VMID=…` override is ignored — each `build-*.sh` re-sources
`config.env`.

**Verify a template before first use** (optional but cheap):

```bash
qm clone 9901 9950 --name verify
qm set 9950 --ipconfig0 ip=10.0.0.50/24,gw=10.0.0.1 --ciuser test --cipassword test
qm start 9950
qm guest exec 9950 -- cloud-init status     # done; sops + age present
qm stop 9950 && qm destroy 9950 --purge
```

## Step 2 — `terraform.tfvars` (base config)

```bash
git clone https://github.com/v2e-sh/v2e-tf.git && cd v2e-tf
```

`terraform.tfvars` is gitignored (it holds secrets). SSH is **key-only** on every
node; the one password you must set, `sudo_password`, is the `v2e` user's **sudo**
password (not an SSH login). The `ansible` automation account has no password
(locked; reached only via `sudo -iu ansible` on control).

```bash
# ---- EDIT THESE ----
PVE_IP=192.168.1.10
PVE_TOKEN='terraform@pve!iac=xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx'
WAN_CIDR='192.168.1.50/24'     # a FREE static IP on the LAN the WAN bridge sits on
WAN_GW='192.168.1.1'           # that LAN's gateway (VyOS's route to the internet)

# v2e sudo password (REQUIRED): a strong one is generated; replace if you prefer.
SUDO_PW=$(openssl rand -base64 12)
echo "v2e sudo password (SAVE THIS): $SUDO_PW"

cat > terraform.tfvars <<EOF
proxmox_endpoint       = "https://${PVE_IP}:8006"
proxmox_api_token      = "${PVE_TOKEN}"
node_name              = "pve"
workstation_public_key = "$(cat ~/.ssh/id_ed25519.pub)"

wan_address = "${WAN_CIDR}"
wan_gateway = "${WAN_GW}"

sudo_password = "${SUDO_PW}"      # v2e sudo password (SSH is key-only)

# Templates at the staging VMIDs from Step 1 (drop these four lines if you
# promoted the templates to the prod IDs 9000-9003):
vyos_template_id   = 9900
ubuntu_template_id = 9901
debian_template_id = 9902
parrot_template_id = 9903
EOF

chmod 600 terraform.tfvars
cat terraform.tfvars                                  # review what's set
```

Worth knowing (all optional — see [`CONFIGURATION.md`](CONFIGURATION.md) for the full
list): `router_password_hash` (break-glass VyOS console password — needs a sha-512
hash from a real OpenSSL 3.x, macOS LibreSSL can't make one), `node_memory = 4096`
(the 2048 default is OOM-tight once the full app estate runs), `trusted_mgmt_sources`,
`agent_egress_allow_tcp_ports`, `ansible_repo_url`/`ansible_repo_ref` (deploy a fork
or feature branch of v2e-ansible).

> **Not on `v2e.sh`?** The app domain is set in **v2e-ansible**
> `inventory/group_vars/services.yml` (`compose_stack_domain`,
> `compose_stack_acme_email`), not in tfvars. Fork v2e-ansible, edit those two
> values, and point `ansible_repo_url` at your fork. See §3 of
> [`CONFIGURATION.md`](CONFIGURATION.md).

> **Rebuilding over a wiped deploy?** A previous plaintext `terraform.tfstate` held
> your mesh SSH private keys + Proxmox/Cloudflare tokens in the clear — treat them as
> **burned**: rotate both tokens before reusing them here, and turn on state
> encryption (Step 4a). The mesh keys regenerate automatically on apply.

## Step 3 — `secrets.sops.yaml` (app secrets, SOPS + age)

One **age-encrypted** secrets file is shipped to control at first boot; cloud-init
decrypts it and passes it to `ansible-playbook` as extra-vars, and the
`compose_stack` role renders the values into each stack's `.env`. **Without it the
app stack does not deploy** (Ansible asserts `cf_dns_api_token`, the `authelia_*`
SSO secrets, and `vaultwarden_admin_token` are set). Design: `specs/2026-06-30-compose-auth-design.md`,
`specs/2026-06-30-compose-traefik-tls-design.md`.

Everything below is copy-paste; the only value you must bring is the **DNS-01
Cloudflare token** from Step 0. Key names are **lowercase** — that's what Ansible
asserts on.

```bash
# ---- EDIT THESE ----
CF_DNS_TOKEN='your-scoped-cloudflare-token'    # Zone:Read + DNS:Edit (Step 0)
ADMIN_USER='admin'                             # Authelia SSO login user
TS_AUTHKEY=''                                  # Tailscale REUSABLE auth key (admin console →
                                               # Settings → Keys); empty = tailnet join skipped
MULLVAD_WG_KEY=''                              # infra mullvad-exit only: WireGuard private key
MULLVAD_WG_ADDRS=''                            # infra mullvad-exit only: WireGuard addresses

# Authelia admin password + argon2id hash. Authelia stores an argon2id hash (not
# bcrypt), so htpasswd no longer applies — the hash comes from the authelia binary.
ADMIN_PASS=$(openssl rand -base64 15)
echo "Authelia ${ADMIN_USER} password (SAVE THIS): $ADMIN_PASS"
ADMIN_HASH=$(docker run --rm authelia/authelia:latest \
  authelia crypto hash generate argon2 --password "$ADMIN_PASS" | sed 's/^Digest: //')

# Authelia SSO scalar secrets + the RS256 OIDC issuer key:
AUTHELIA_SESSION=$(openssl rand -base64 32)
AUTHELIA_STORAGE=$(openssl rand -base64 32)
AUTHELIA_RESET_JWT=$(openssl rand -base64 32)
AUTHELIA_OIDC_HMAC=$(openssl rand -base64 32)
OIDC_ISSUER_KEY=$(openssl genrsa 4096 2>/dev/null)

# Per-app OIDC client secrets — plaintext for the app (.env) + pbkdf2 hash for Authelia:
ARCANE_OIDC_SECRET=$(openssl rand -base64 24)
ARCANE_OIDC_HASH=$(docker run --rm authelia/authelia:latest \
  authelia crypto hash generate pbkdf2 --variant sha512 --password "$ARCANE_OIDC_SECRET" | sed 's/^Digest: //')
GRAFANA_OIDC_SECRET=$(openssl rand -base64 24)
GRAFANA_OIDC_HASH=$(docker run --rm authelia/authelia:latest \
  authelia crypto hash generate pbkdf2 --variant sha512 --password "$GRAFANA_OIDC_SECRET" | sed 's/^Digest: //')

VAULTWARDEN_ADMIN=$(openssl rand -base64 32)

GRAFANA_PW=$(openssl rand -base64 15)
SEMAPHORE_PW=$(openssl rand -base64 15)
TECHNITIUM_PW=$(openssl rand -base64 15)
RUSTDESK_PW=$(openssl rand -base64 15)
echo "Grafana admin (SAVE THIS):        $GRAFANA_PW"
echo "Semaphore admin (SAVE THIS):      $SEMAPHORE_PW"
echo "Technitium admin (SAVE THIS):     $TECHNITIUM_PW"
echo "RustDesk unattended (SAVE THIS):  $RUSTDESK_PW"

# 1. Author the plaintext secrets file (deleted after encryption):
cat > secrets.yaml <<EOF
cf_dns_api_token: "${CF_DNS_TOKEN}"
authelia_session_secret: "${AUTHELIA_SESSION}"
authelia_storage_encryption_key: "${AUTHELIA_STORAGE}"
authelia_reset_password_jwt_secret: "${AUTHELIA_RESET_JWT}"
authelia_oidc_hmac_secret: "${AUTHELIA_OIDC_HMAC}"
authelia_admin_password_hash: "${ADMIN_HASH}"
authelia_oidc_arcane_client_secret_hash: "${ARCANE_OIDC_HASH}"
authelia_oidc_grafana_client_secret_hash: "${GRAFANA_OIDC_HASH}"
authelia_oidc_arcane_client_secret: "${ARCANE_OIDC_SECRET}"
authelia_oidc_grafana_client_secret: "${GRAFANA_OIDC_SECRET}"
authelia_oidc_issuer_key: |
$(printf '%s\n' "${OIDC_ISSUER_KEY}" | sed 's/^/  /')
vaultwarden_admin_token: "${VAULTWARDEN_ADMIN}"
semaphore_db_pass: "$(openssl rand -base64 18)"
semaphore_admin_password: "${SEMAPHORE_PW}"
semaphore_access_key_encryption: "$(openssl rand -base64 32)"
semaphore_runner_reg_token: "$(openssl rand -base64 24)"
arcane_encryption_key: "$(openssl rand -base64 32)"
arcane_jwt_secret: "$(openssl rand -base64 32)"
grafana_admin_password: "${GRAFANA_PW}"
technitium_admin_password: "${TECHNITIUM_PW}"
tailscale_authkey: "${TS_AUTHKEY}"
rustdesk_unattended_password: "${RUSTDESK_PW}"
mullvad_wireguard_private_key: "${MULLVAD_WG_KEY}"
mullvad_wireguard_addresses: "${MULLVAD_WG_ADDRS}"
EOF

# 2. Age key control uses to decrypt (shipped to control by cloud-init):
age-keygen -o keys.txt
AGE_PUB=$(grep -oE 'age1[0-9a-z]+' keys.txt | head -1)

# 3. Backup/escrow key — keep keys-backup.txt OFFLINE, never on control.
#    It can also decrypt, so a lost keys.txt is recoverable, not fatal.
age-keygen -o keys-backup.txt
AGE_BACKUP=$(grep -oE 'age1[0-9a-z]+' keys-backup.txt | head -1)

# 4. Encrypt to BOTH recipients, then delete the plaintext:
sops --encrypt --age "${AGE_PUB},${AGE_BACKUP}" secrets.yaml > secrets.sops.yaml
rm -P secrets.yaml

cat >> terraform.tfvars <<EOF

sops_secrets_file = "./secrets.sops.yaml"   # -> control:~ansible/ansible/group_vars/all.sops.yaml
sops_age_key_file = "./keys.txt"            # -> control:~/.config/sops/age/keys.txt
EOF

chmod 600 terraform.tfvars keys.txt keys-backup.txt secrets.sops.yaml
```

What each key is for, and the exact generation rule per key: §2 of
[`CONFIGURATION.md`](CONFIGURATION.md).
Edit values later with `sops secrets.sops.yaml`; rotate recipients with
`sops updatekeys` after editing `.sops.yaml`.

> `keys.txt` and `keys-backup.txt` decrypt **everything**. Both are gitignored, but
> prefer moving them out of the repo tree; keep the backup offline. The age private
> key is also read into tf state — one more reason for Step 4a.

## Step 4 — Optional blocks

### 4a. Encrypt the OpenTofu state (recommended)

State is **plaintext by default** and holds the mesh SSH private keys, the Proxmox
and Cloudflare tokens, `sudo_password`, and the age key — any copy of
`terraform.tfstate` is a full compromise. Encryption is opt-in via the
`TF_ENCRYPTION` env var; skipping it just prints a WARNING each run.

```bash
security add-generic-password -a "$USER" -s tofu-v2e-state -w "$(openssl rand -base64 32)" 2>/dev/null
echo "BACK UP THIS PASSPHRASE: $(security find-generic-password -a "$USER" -s tofu-v2e-state -w)"

export TF_ENCRYPTION="$(cat <<EOF
key_provider "pbkdf2" "main" { passphrase = "$(security find-generic-password -a "$USER" -s tofu-v2e-state -w)" }
method "aes_gcm" "main" { keys = key_provider.pbkdf2.main }
state { method = method.aes_gcm.main }
plan  { method = method.aes_gcm.main }
EOF
)"
```

> Once state is encrypted, `TF_ENCRYPTION` must be exported in **every** shell before
> any tofu command; lose the passphrase and the state is unrecoverable. Migrating an
> already-deployed plaintext state: see the appendix at the bottom.

### 4b. Cloudflare SSH tunnel to control

A WAN-independent SSH path to control (`cloudflared` is installed on control at
first boot; best-effort — a tunnel hiccup never fails the deploy). Uses the
**tunnel token** from Step 0. Skip this block and the lab deploys unchanged;
teardown is clean either way (`tofu destroy` removes the tunnel + CNAME).

```bash
# ---- EDIT THESE ----
CF_API_TOKEN='cf_tunnel_scoped_token'
CF_ACCOUNT_ID='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_ZONE_ID='xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx'
CF_DOMAIN='example.com'        # a zone already on your Cloudflare account
CF_SUBDOMAIN='lab'             # public host becomes lab.example.com
CLOUDFLARE_ACCESS_EMAIL=""     # optional OTP gate before SSH; empty = no gate

cat >> terraform.tfvars <<EOF

cloudflare_api_token  = "${CF_API_TOKEN}"
cloudflare_account_id = "${CF_ACCOUNT_ID}"
cloudflare_zone_id    = "${CF_ZONE_ID}"
tunnel_hostname       = "${CF_SUBDOMAIN}.${CF_DOMAIN}"
tunnel_dns_name       = "${CF_SUBDOMAIN}"
EOF
[ -n "$CLOUDFLARE_ACCESS_EMAIL" ] && echo "cloudflare_access_emails = [\"$CLOUDFLARE_ACCESS_EMAIL\"]" >> terraform.tfvars

chmod 600 terraform.tfvars
```

## Step 5 — Deploy

> **Preflight:** `ssh-add -l` shows your key; `ssh root@<PVE> true` returns silently;
> if you did 4a, `TF_ENCRYPTION` is exported in this shell.

```bash
tofu init        # first time only
tofu plan        # review: 2 keypairs + router (+snippet) + 4 nodes (+snippets) + time_sleep
tofu apply       # type 'yes'
```

`apply` creates the router first, waits `router_boot_wait` (default 120s) for VyOS
to boot and apply routing, then builds the four nodes. Control's first boot then
clones v2e-ansible and runs `site.yml` across the mesh (phases: bootstrap/hardening →
services/Docker/compose → applications → infra → tailscale → control-desktop) — allow **10–20 minutes** after apply
finishes before judging the result.

## Step 6 — Watch it converge, then verify

Host keys change on every rebuild — clear stale ones, then SSH in. If your mac can't
route to the WAN subnet, jump through the PVE host (it's dual-homed):

```bash
WAN_IP=192.168.1.50                                  # host part of wan_address
ssh-keygen -R "[$WAN_IP]:2201"
ssh -J root@192.168.1.10 -p 2201 v2e@"$WAN_IP"       # -p applies to the final hop
```

**1. Cloud-init + Ansible finished on control:**

```bash
v2e@control:~$ sudo cloud-init status --long                      # → done
v2e@control:~$ sudo grep -A5 'PLAY RECAP' /var/log/cloud-init-output.log   # failed=0 on every host
```

**2. Both meshes work from control** (private keys live only on control):

```bash
v2e@control:~$ ssh services hostname && ssh agent hostname && ssh infra hostname
v2e@control:~$ sudo -iu ansible ssh services hostname             # automation mesh, NOPASSWD sudo
```

**3. The app estate is up on services:**

```bash
v2e@control:~$ ssh services docker ps --format '{{.Names}}\t{{.Status}}'
# expect: traefik, authelia, whoami, semaphore(+postgres), arcane, vaultwarden,
#         prometheus, grafana, loki, alloy, uptime-kuma, node-exporter, cadvisor — all Up (healthy)
```

**4. The wildcard cert was issued** (DNS-01 — no inbound ports needed):

```bash
v2e@control:~$ ssh services docker logs traefik 2>&1 | grep -i 'certificates obtained'
# "Certificates obtained for domains [int.<domain> *.int.<domain>]"
```

If issuance fails or stalls, do **not** add local DNS propagation checks or pin
resolvers — see the DNS-01 note in Troubleshooting.

**5. Apps answer over HTTPS.** DNS for `int.<domain>` isn't set up yet at this point
(Step 8), so pin the name to the services node with `--resolve`:

```bash
v2e@control:~$ curl -s --resolve whoami.int.example.com:443:10.1.2.10 https://whoami.int.example.com | head -3
v2e@control:~$ curl -sI --resolve grafana.int.example.com:443:10.1.2.10 https://grafana.int.example.com | head -1
# whoami: plaintext request dump, valid cert (no -k needed) · grafana: redirect to Authelia
```

**6. If you enabled SOPS-verify it decrypts on control:**

```bash
v2e@control:~$ sudo -iu ansible sops -d ~/ansible/group_vars/all.sops.yaml | head -2
```

## Step 7 — Reach the lab day-to-day

- **SSH, via WAN DNAT:** `ssh -J root@<pve> -p 2201 v2e@<wan-ip>` (or directly
  `ssh -p 2201 v2e@<wan-ip>` if your machine routes to the WAN subnet).
- **SSH, via Cloudflare tunnel** (if Step 4b):
  ```bash
  brew install cloudflared
  ssh -o ProxyCommand="cloudflared access ssh --hostname lab.example.com" v2e@lab.example.com
  ```
- **Everything inside:** from control — `ssh services|agent|infra|vyos` as `v2e`;
  `sudo -iu ansible` for the automation mesh. `tofu output` prints all connection
  hints.
- **Browser access to the apps from your machine** requires internal DNS + a route —
  that's Step 8 (Technitium + Tailscale), currently manual.

### Using the installed apps (simple setup)

Once you're on the tailnet with split-DNS (Step 8), the apps are three steps away:

1. **Reach them.** Everything is at `*.int.example.com` (e.g. `https://grafana.int.example.com`),
   resolvable only over Tailscale. If a name doesn't resolve, check you're not on an
   exit node that shadows the split-DNS, or add `/etc/resolver/int.example.com` (macOS).
2. **Log in once at Authelia.** Any app bounces you to `auth.int.example.com`; sign in as
   `admin`. That one session covers every SSO app — Grafana, Semaphore, Arcane, Traefik,
   Uptime Kuma, ntfy. (Grafana/Arcane show a "Sign in with Authelia/OIDC" button.)
3. **The two non-SSO bits:** point the **Bitwarden** client at `https://vault.int.example.com`
   for the password vault, and subscribe to the ntfy topic **`v2e-alerts`** for alerts.

Full per-app guide (what each app is for, how to log in): **[Using the lab](guides/using-the-apps.md)**.

Sudo as `v2e` prompts for `sudo_password` from Step 2. `root` SSH is disabled
everywhere; `ansible` is internal-only.

## Step 8 — Post-deploy steps that stay manual

The Technitium stacks + zone, Tailscale join, RustDesk client, and control-desktop
settings are all Ansible-managed (phases 04/05/06; design:
`specs/2026-06-30-dns-appliance-design.md`). What remains is genuinely manual —
tailnet-side settings and one client-side pin:

1. **Tailscale admin console** (after the first converge with a `tailscale_authkey`
   set): **approve** control's advertised route `10.1.0.0/16`, and set **split-DNS**
   `int.<domain>` → `10.1.0.10`. That gives your mac browser access to every app.
   (macOS: if split-DNS is flaky, `echo 'nameserver 10.1.0.10' | sudo tee
   /etc/resolver/int.<domain>` — and note Mullvad hijacks DNS while connected.)
2. **RustDesk release pin** — set `rustdesk_client_version` + the `.deb` sha256 in
   v2e-ansible `inventory/group_vars/control.yml` (the role skips with a warning
   until pinned). Then connect from your mac via **Direct IP** to control's tailnet
   IP (the public-ID path requires RustDesk's login now); the unattended password is
   `rustdesk_unattended_password` from Step 3.
3. **Skipped something in Step 3?** An empty `tailscale_authkey` or RustDesk pin
   just skips those phases — set the value and re-run Ansible (Day-2 section) to
   pick them up.

## Day-2 operations

```bash
tofu apply -replace='proxmox_virtual_environment_vm.vyos'              # rebuild just the router
tofu apply -replace='proxmox_virtual_environment_vm.node["agent"]'    # rebuild one node
tofu destroy                                                          # tear it all down (removes tunnel + CNAME too)
```

- cloud-init runs **once** per VM: tfvars changes do *not* re-apply to a live VM —
  `-replace` the affected VM.
- Re-run Ansible any time from control: `sudo -iu ansible bash -lc 'cd ~/ansible &&
  ansible-playbook -i inventory/hosts.ini site.yml -e @~/.v2e-secrets.yml'`.
- Operational playbooks (never auto-run): `playbooks/ops/patch.yml`,
  `ops/killswitch.yml` (tag-gated), `ops/vyos-hardening.yml`, `ops/agents.yml`.
- **Firewall changes:** the VyOS default-deny policy is defined in Terraform
  (`cloud-init/vyos-router.yaml.tftpl`) and applied at router build — edit, then
  rebuild the router with `-replace`. Live `set firewall` edits are wiped on the next
  rebuild. Knobs: `firewall_enabled`, `trusted_mgmt_sources`,
  `agent_egress_restricted`, `agent_egress_allow_tcp_ports`.
- **DR:** state backup, age-key rotation/escrow, snapshot restore = the DOCS-2
  runbook (not yet written). Keep `keys-backup.txt` offline until then.

## Troubleshooting & hard-won gotchas (don't re-learn these)

**Terraform / deploy**
- **`Unsupported argument` on apply** → a tfvars key that isn't a real variable. The
  password vars are exactly `sudo_password` and `router_password_hash`.
- **`sudo_password` is required** — omitting it, or a known-weak value
  (`v2e`/`ansible`/`password`/`changeme`), fails validation.
- **`unable to find configuration file for VM 9000`** → template VMID doesn't exist:
  build it, or point the `*_template_id` vars at the staging `9900–9903`.
- **`tofu apply` SSHes to the PVE node** to upload snippets. Wrong network / key not
  in agent = "unreachable" failures.
- **Teardown hangs on `Still destroying…`** → bpg waits on the guest agent. VMs set
  `stop_on_destroy = true`; if one still wedges: `qm stop <id> --skiplock && qm
  destroy <id> --purge --skiplock`, then `tofu state rm` it.
- **`plan`/`apply` stuck on `Refreshing…`** → an unresponsive guest agent; use
  `tofu plan -refresh=false`.
- **Cloudflare `404 Tunnel not found`** → tunnel deleted out-of-band; `tofu state rm`
  the `cloudflare_*` resources and re-apply.

**Passwords & hashes**
- **Authelia admin hash is argon2id** — generated with the authelia binary (Step 3),
  not `htpasswd`; the old TinyAuth `$2y$`→`$2a$` bcrypt rewrite no longer applies.
- **Router hash needs OpenSSL 3.x** — macOS LibreSSL has no `passwd -6`; use
  `$(brew --prefix openssl@3)/bin/openssl passwd -6`, or leave key-only (default).
- **`$` in any secret gets eaten by docker compose interpolation** — the Ansible
  `env.j2` escapes every secret as `$$`; a hand-written `.env` must do the same.

**First boot / Ansible**
- **Nodes' apt step fails on first boot** → VyOS wasn't routing yet; bump
  `router_boot_wait`. Login still works (users/keys are set before packages).
- **First-boot apt upgrade is OFF by default** (`package_upgrade = false`) — it can
  OOM control and hang the apply. Ansible owns patching.
- **cloudflared on ParrotOS**: Parrot's codename isn't served by the cloudflared apt
  repo — cloud-init uses the `any` suite and the install is best-effort by design.
  The tunnel is a convenience path, never a dependency.
- **`compose_stack` asserts secrets are empty** in a full unattended run → secrets
  must arrive as extra-vars (`-e @file`), which cloud-init wires up; group_vars-only
  resolution is cleared by `meta: reset_connection` in the docker role.

**Certificates / DNS**
- **DNS-01 self-checks self-poison on this lab's network** — the WAN path runs a
  caching transparent `:53` interceptor, and lego's pre-flight lookup negative-caches
  the challenge name for 1800s. Traefik therefore runs with
  `propagation.delayBeforeChecks=60s` + `disableChecks=true` (Let's Encrypt validates
  from *its* resolvers). **Never** enable local checks or pin `resolvers=1.1.1.1`.
- **Any public A/AAAA record for an internal name must be gray-cloud** (DNS-only);
  orange-cloud serves Cloudflare's edge cert instead.

**Access**
- **Host keys change on every recreate** → `ssh-keygen -R` first (also
  `ssh-keygen -R lab.<domain>` after a rebuild when using the tunnel).
- **VyOS needs a cloud-init-built image**; router login (keys/password) goes in
  top-level cloud-config, not `vyos_config_commands` (VyOS cloud-init never commits).
- **`wan_gateway` is whatever NATs the WAN bridge to the internet** — your LAN
  gateway/PVE host, not a separate router.
- **Pasting into zsh:** keep tokens in single quotes (`!` triggers history
  expansion); the heredocs here are paste-safe.
- **SOPS "half-configured" check** → set both `sops_secrets_file` and
  `sops_age_key_file`, or neither. If control can't decrypt, the file wasn't
  encrypted to the key in `keys.txt`.
- **The mesh private keys are embedded in control's cloud-init drive** — treat that
  VM's disk/snapshots as key material.

## Appendix — migrating an existing plaintext state to encrypted

Only needed once, if you deployed before enabling Step 4a. Adds a temporary
`unencrypted` fallback so the first read loads the plaintext, then rewrites it
encrypted.

```bash
cp terraform.tfstate terraform.tfstate.preencrypt.bak
export TF_ENCRYPTION="$(cat <<EOF
key_provider "pbkdf2" "main" { passphrase = "$(security find-generic-password -a "$USER" -s tofu-v2e-state -w)" }
method "aes_gcm" "main" { keys = key_provider.pbkdf2.main }
method "unencrypted" "migrate" {}
state {
  method = method.aes_gcm.main
  fallback { method = method.unencrypted.migrate }
}
plan {
  method = method.aes_gcm.main
  fallback { method = method.unencrypted.migrate }
}
EOF
)"
tofu apply                                   # rewrites state ENCRYPTED
head -c 120 terraform.tfstate                # confirm: encryption metadata, not readable attrs

# Re-export the NORMAL form (Step 4a, no fallback), then:
tofu plan                                    # reads encrypted, zero drift
rm -P terraform.tfstate.backup terraform.tfstate.preencrypt.bak
```

- `TF_ENCRYPTION` must be set for **every** command after migration (`init` included).
- Don't rename the `pbkdf2 "main"` / `aes_gcm "main"` labels — ciphertext metadata is
  keyed to them.
- `terraform.tfstate.backup` stays plaintext after migration → shred it (above).
